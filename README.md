# CloudPlush — PWA Push Notification Micro-App

A self-hosted Progressive Web App that receives real push notifications on iOS (and Android/desktop). You send notifications from a CLI tool or a web admin panel, and they arrive as native OS-level notifications on subscribed devices.

Live at: `https://notifications.cloudplush.com`

## How It Works (The Big Picture)

```
You (CLI / Admin Page)
        |
        |  POST /api/send  {"title": "...", "body": "..."}
        v
  +-----------+        VAPID-signed push
  |  FastAPI  | --------------------------> Apple/Google/Mozilla Push Service
  |  server   |                                       |
  +-----------+                                       |
        ^                                             | push event
        |  subscribe (one-time)                       v
        |                                     +-------------+
        +-------------------------------------|  iPhone PWA  |
           stores push subscription           |  (sw.js)     |
           in SQLite                          +-------------+
                                                    |
                                              OS notification
                                              appears on phone
```

**The flow in plain English:**

1. The PWA is installed on a phone and the user taps "Enable Notifications."
2. The browser generates a **push subscription** (an endpoint URL + encryption keys) and sends it to the server.
3. The server stores that subscription in SQLite.
4. When you want to send a notification, you call `POST /api/send` (from the CLI, admin page, or any HTTP client).
5. The server uses `pywebpush` to sign the message with the VAPID private key and POST it to the push service (Apple's APNs for iOS, Google's FCM for Chrome, Mozilla's autopush for Firefox).
6. The push service delivers the message to the device.
7. The service worker (`sw.js`) running in the background receives it and calls `showNotification()`.

## First-Time Setup

This only needs to be done once, on the machine where the repo lives.

### 1. Install Python dependencies (for setup.py and CLI)

```bash
cd ios_pwa
pip install -r requirements.txt
```

### 2. Generate VAPID keys and icons

```bash
python setup.py
```

This creates:
- `vapid_keys.json` — your ECDSA P-256 key pair (keep this safe, back it up)
- `static/icon-192.png` and `static/icon-512.png` — purple "C" app icons

If `vapid_keys.json` already exists, it skips key generation (safe to re-run).

### 3. Create the `.env` file

```bash
echo CLOUDFLARED_TOKEN=your_tunnel_token_here > .env
```

The token is the long base64 string from your Cloudflare Zero Trust dashboard.

### 4. Configure the Cloudflare tunnel

In the [Cloudflare Zero Trust dashboard](https://one.dash.cloudflare.com/), configure your tunnel's public hostname:

| Field | Value |
|-------|-------|
| Subdomain | `notifications` |
| Domain | `cloudplush.com` |
| Service type | `HTTP` |
| URL | `app:8080` |

The URL `app:8080` refers to the Docker Compose service name `app` on its internal port. The two containers share a Docker network called `tunnel`, so `cloudflared` can reach `app` by name.

## Running with Docker Compose

```bash
cd ios_pwa

# Build and start both containers
docker compose up -d

# Check they're running
docker compose ps

# View app logs
docker compose logs -f app

# View tunnel logs
docker compose logs -f cloudflared

# Stop everything
docker compose down

# Rebuild after code changes
docker compose build app && docker compose up -d
```

### What the compose file does

```yaml
services:
  app:          # FastAPI server
    build: .
    ports:
      - "8899:8080"     # Host port 8899 -> container port 8080
    volumes:
      - ./vapid_keys.json:/app/vapid_keys.json:ro   # Mount keys read-only
      - ./data:/app/data                             # Persist SQLite DB
    networks:
      - tunnel

  cloudflared:   # Cloudflare tunnel sidecar
    image: cloudflare/cloudflared:latest
    command: tunnel --no-autoupdate run --token ${CLOUDFLARED_TOKEN}
    networks:
      - tunnel  # Same network as app, so it can reach http://app:8080
```

- The app is available at `http://localhost:8899` on the host machine.
- The Cloudflare tunnel exposes it publicly at `https://notifications.cloudplush.com`.
- `vapid_keys.json` is mounted read-only (`:ro`) — the container never writes to it.
- `./data` is mounted so the SQLite database survives container restarts.

## Running Without Docker (Dev Mode)

For local development without Docker:

```bash
cd ios_pwa
pip install -r requirements.txt
python setup.py            # first time only
uvicorn server:app --host 0.0.0.0 --port 8899 --reload
```

The `--reload` flag watches for file changes. The app is at `http://localhost:8899`.

Note: Push notifications require HTTPS, so they won't work from `localhost` on a phone. You still need the Cloudflare tunnel (or another HTTPS reverse proxy) for the phone to subscribe.

## Installing the PWA on iPhone

iOS supports Web Push only for PWAs added to the Home Screen (iOS 16.4+).

### Step-by-step:

1. Open **Safari** on iPhone and go to `https://notifications.cloudplush.com`
2. Tap the **Share** button (square with arrow)
3. Scroll down and tap **Add to Home Screen**
4. Tap **Add** (you can rename it if you want)
5. Close Safari. Open the app **from the Home Screen icon** (this is critical — push only works in standalone mode, not in Safari)
6. Enter a device name (or keep the auto-detected "iPhone")
7. Tap **Enable Notifications**
8. iOS will prompt "notifications.cloudplush.com would like to send you notifications" — tap **Allow**

You're subscribed. The status should show "Subscribed" in green.

### Why it must be opened from the Home Screen

Safari doesn't support the Push API directly. Only "standalone" PWAs (opened from the Home Screen icon) get push capabilities. The `manifest.json` sets `"display": "standalone"` which tells iOS to run it as an app, not a browser tab.

## Sending Notifications

### From the Admin Page

Go to `https://notifications.cloudplush.com/admin` (or `http://localhost:8899/admin`).

- Fill in the **title** and **body**
- Device checkboxes appear below — all checked by default. Uncheck to exclude specific devices.
- Click **Send Notification**
- A green "Sent to N device(s)" confirmation appears
- The notification log at the bottom refreshes automatically

### From the CLI

```bash
# Send to all devices
python send_notify.py -t "Server Alert" -b "Disk usage at 90%"

# Send to specific device IDs
python send_notify.py -t "Hello" -b "Just for you" -d 1

# Send to multiple specific devices
python send_notify.py -t "Team" -b "Standup in 5" -d 1 3

# Use a different server URL
python send_notify.py -t "Test" -b "Hello" -s http://some-other-host:8899
```

The default server URL is `http://localhost:8899`.

### From any HTTP client (curl, scripts, etc.)

```bash
# Send to all devices
curl -X POST http://localhost:8899/api/send \
  -H "Content-Type: application/json" \
  -d '{"title": "Hello", "body": "From curl"}'

# Send to specific devices
curl -X POST http://localhost:8899/api/send \
  -H "Content-Type: application/json" \
  -d '{"title": "Hello", "body": "Targeted", "device_ids": [1, 2]}'
```

Response: `{"sent": 1, "failed": 0}`

## Architecture Deep Dive

### Backend: `server.py`

A single-file FastAPI app. No ORM, no migrations framework — just raw SQLite and `html.escape()` for output. Intentionally minimal for a PoC.

**Key design decisions:**

- **Dual content-type handling on `/api/send`**: Accepts both `application/json` (from CLI/curl) and `application/x-www-form-urlencoded` (from HTMX forms). Checks `Content-Type` header to decide how to parse.
- **HTMX-aware responses**: If the request has an `HX-Request` header (set automatically by HTMX), the server returns HTML fragments. Otherwise, it returns JSON. This lets the same endpoint serve both the admin page and the API.
- **`HX-Trigger` header**: After a successful send, the response includes `HX-Trigger: notificationSent`. HTMX on the admin page listens for this event to refresh the notification log and device list without polling.
- **Service worker at root scope**: `sw.js` is served from `/sw.js` (not `/static/sw.js`) so its scope covers the entire origin. A service worker can only control pages within its scope.

### Frontend: HTMX + Vanilla JS

- **HTMX** handles all server communication (form submission, polling, live refresh).
- **Vanilla JS** handles the Push API (subscribe/unsubscribe) because HTMX can't call browser APIs.
- **BroadcastChannel** bridges the service worker and the page: when a push arrives while the page is open, the service worker posts a message, and the page triggers an HTMX refresh.
- **No build step**, no npm, no bundler. Just static HTML files.

### Service Worker: `sw.js`

Two event listeners:
- **`push`**: Receives the notification payload from the push service, calls `showNotification()` to display it, and posts to the `BroadcastChannel` so open pages refresh.
- **`notificationclick`**: When the user taps the notification, focuses an existing app window or opens a new one.

The service worker is installed with `skipWaiting()` and `clients.claim()` so updates take effect immediately without needing to close all tabs.

### Database: SQLite

Two tables (see [Database](#database) section below). The DB file lives at `data/notifications.db` and is created automatically on first server start.

## VAPID Keys Explained

VAPID = **V**oluntary **A**pplication **S**erver **Id**entification.

### What they are

An ECDSA P-256 key pair stored in `vapid_keys.json`:

```json
{
  "privateKey": "base64url-encoded-32-bytes",
  "publicKey": "base64url-encoded-65-bytes"
}
```

### Where each key is used

| Key | Where | Purpose |
|-----|-------|---------|
| **Public key** | Sent to the browser during `pushManager.subscribe()` as the `applicationServerKey` | Tells the push service: "only accept messages signed by the holder of this key's private counterpart." Also served at `GET /api/vapid-public-key`. |
| **Private key** | Used server-side by `pywebpush` when sending a notification | Signs each push message so Apple/Google/Mozilla can verify it came from the legitimate application server. Never leaves the server. |

### Why they exist

Without VAPID, anyone who discovered a push subscription endpoint URL could spam notifications to that device. VAPID creates a cryptographic binding: the subscription is locked to your public key, and only someone with the matching private key can send to it.

### Are they secret?

- **Public key**: No. It's public by design — every subscribing browser receives it.
- **Private key**: **Yes.** If someone gets it, they can send push notifications to all your subscribers. Treat it like a TLS private key.

### When to regenerate

Almost never. Regenerate only if:

1. **The private key is leaked/compromised** — someone unauthorized could send pushes.
2. **You want to start completely fresh** — new identity, new subscribers.

### How to regenerate

```bash
# 1. Delete the old keys
del vapid_keys.json        # Windows
rm vapid_keys.json         # Linux/Mac

# 2. Regenerate
python setup.py

# 3. Rebuild and restart
docker compose build app && docker compose up -d

# 4. On EVERY subscribed device:
#    Open the PWA -> Disable Notifications -> Enable Notifications
#    (old subscriptions are permanently invalid after key change)
```

### What happens to existing subscribers after regeneration

They break. Each push subscription is cryptographically bound to the public key that was active when the device subscribed. After you regenerate keys, the push service will reject messages signed with the new private key for old subscriptions (they'll return HTTP 401 or 403). Every device must re-subscribe.

## Cloudflare Tunnel Setup

The Cloudflare tunnel connects `notifications.cloudplush.com` to the Docker container running on your local machine. No port forwarding, no static IP needed.

### How it works

1. `cloudflared` (running as a Docker container) establishes an outbound-only connection to Cloudflare's edge.
2. When someone visits `notifications.cloudplush.com`, Cloudflare routes the request through the tunnel to `cloudflared`.
3. `cloudflared` forwards it to `http://app:8080` (the FastAPI container on the shared Docker network).
4. The response travels back through the tunnel.

### The token

The `CLOUDFLARED_TOKEN` in `.env` is a JWT that identifies your tunnel. It's generated in the Cloudflare Zero Trust dashboard and contains your account ID and tunnel ID. It doesn't expire but can be revoked from the dashboard.

### Dashboard configuration

In [Cloudflare Zero Trust](https://one.dash.cloudflare.com/) > Access > Tunnels > your tunnel > Public Hostname:

- Subdomain: `notifications`
- Domain: `cloudplush.com`
- Service: `HTTP`
- URL: `app:8080`

The URL `app:8080` works because both containers share the `tunnel` Docker network. Docker's internal DNS resolves `app` to the FastAPI container's IP.

## Database

SQLite database at `data/notifications.db`. Created automatically on first server start.

### Tables

**`subscriptions`** — one row per subscribed device.

| Column | Type | Description |
|--------|------|-------------|
| `id` | INTEGER PK | Auto-incrementing device ID (used for targeting) |
| `endpoint` | TEXT UNIQUE | Push service URL (e.g., `https://web.push.apple.com/...`) |
| `sub_json` | TEXT | Full subscription JSON (endpoint + encryption keys) |
| `device_name` | TEXT | User-provided label (e.g., "iPhone", "iPad") |
| `created_at` | TIMESTAMP | When the device subscribed |

**`notifications`** — log of all sent notifications.

| Column | Type | Description |
|--------|------|-------------|
| `id` | INTEGER PK | Auto-incrementing |
| `title` | TEXT | Notification title |
| `body` | TEXT | Notification body |
| `sent_at` | TIMESTAMP | When it was sent |

### Backup

The database is a single file. To back up:

```bash
copy data\notifications.db data\notifications.db.bak
```

### Reset

To wipe everything (subscriptions + notification history):

```bash
del data\notifications.db
docker compose restart app
```

Devices will need to re-subscribe.

## API Reference

All endpoints are on the FastAPI app (port 8080 inside Docker, 8899 on host).

### Pages

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | PWA main page (subscribe + notification log) |
| GET | `/admin` | Admin page (send notifications, pick devices) |
| GET | `/sw.js` | Service worker (served at root scope) |
| GET | `/manifest.json` | PWA manifest |

### JSON API

| Method | Path | Body | Response |
|--------|------|------|----------|
| GET | `/api/vapid-public-key` | — | `{"publicKey": "..."}` |
| POST | `/api/subscribe` | `{"subscription": {...}, "device_name": "iPhone"}` | `{"ok": true}` |
| POST | `/api/unsubscribe` | `{"endpoint": "https://..."}` | `{"ok": true}` |
| POST | `/api/send` | `{"title": "...", "body": "...", "device_ids": [1,2]}` | `{"sent": 1, "failed": 0}` |
| GET | `/api/subscriptions/count` | — | `{"count": 2}` |

### HTMX Endpoints (return HTML fragments)

| Method | Path | Returns |
|--------|------|---------|
| GET | `/api/notifications` | HTML list of notification items |
| GET | `/api/devices` | HTML checkboxes for each subscribed device |
| GET | `/api/subscriptions/count` | HTML text like "2 devices subscribed" |
| POST | `/api/send` | HTML success/error message (when `HX-Request` header is present) |

The `/api/send` endpoint is dual-mode: it checks the `Content-Type` header to parse JSON or form data, and checks the `HX-Request` header to decide whether to return HTML or JSON.

## Troubleshooting

### "Push API not supported" on iPhone

You're opening it in Safari, not from the Home Screen. The Push API is only available in standalone PWA mode. Close Safari, open the app from the Home Screen icon.

### iPhone says "Enable Notifications" but nothing happens when tapped

Check that iOS is 16.4 or later (Settings > General > About > iOS Version). Web Push was added in 16.4.

### Admin page form doesn't submit

Check the browser console for JavaScript errors. HTMX is bundled locally at `/static/htmx.min.js` — if this file is missing from the Docker image, the form won't work (it relies on HTMX for AJAX submission). Rebuild the image:

```bash
docker compose build app && docker compose up -d
```

### Notification sent successfully but device doesn't receive it

- The device might have notifications disabled in iOS Settings > Notifications > CloudPlush.
- The push subscription might have expired. The server auto-cleans 410 (Gone) subscriptions, but stale ones can linger. Have the device unsubscribe and re-subscribe.
- Check `docker compose logs app` for `WebPushException` errors.

### "vapid_keys.json not found" on server start

Run `python setup.py` first, or ensure the volume mount in `docker-compose.yml` points to the correct path.

### Cloudflared tunnel not connecting

```bash
docker compose logs cloudflared
```

Look for authentication errors (bad token) or DNS errors. Verify the token in `.env` matches your tunnel in the Cloudflare dashboard.

### Database is locked errors

SQLite doesn't handle concurrent writes well. This shouldn't happen in normal use (single-threaded uvicorn), but if you're running multiple workers, switch to WAL mode or use PostgreSQL.

## Security Considerations

This is a proof-of-concept. For production use, consider:

- **Authentication on `/admin` and `/api/send`**: Currently open. Anyone who can reach the server can send notifications. Add an API key, basic auth, or put it behind Cloudflare Access.
- **Rate limiting**: No rate limiting on any endpoint. A malicious actor could spam notifications.
- **Input validation**: Title and body are escaped for HTML output but not length-limited.
- **HTTPS only**: The Cloudflare tunnel handles HTTPS. If you expose port 8899 directly, it's unencrypted HTTP.
- **VAPID private key**: Stored in a plain JSON file on disk. For production, use a secrets manager or environment variable.
- **SQLite**: Fine for a handful of devices. For more, consider PostgreSQL.
